# 大学官网同步状态前端接线设计

**日期：** 2026-05-21  
**范围：** 在运营情报 dashboard 中，把 `university_site` 的真实同步状态接到前端，并明确展示 `success / blocked / error`。  
**阶段：** 小范围前端接线，不改整体信息架构。

---

## 目标

当前大学官网链路已经有真实数据和同步状态，但前端还停留在“列表展示”阶段，用户看不到：

- 哪些学校本轮抓取成功
- 哪些学校被反爬拦截
- 总览页的来源状态是否真实健康

这次改动的目标是把这层可观测性直接放进现有 dashboard，而不是再造一套管理后台。

---

## 决策

1. 保持现有导航和卡片流不变，只补状态层
2. `university_site` 来源页直接展示逐学校同步状态卡
3. 总览页顶部“来源状态”改为真实大学官网同步健康度
4. 状态接口失败时，总览页降级为 `待同步`，不拖垮整页

---

## 数据流

### 总览页

- 继续读取 `/api/intel/overview`
- 并行追加读取 `/api/intel/sources/university_site/sync-status`
- 用同步状态计算顶部 `来源状态`

### 大学官网来源页

- 继续读取 `/api/intel/sources/university_site`
- 直接使用响应中的 `sync_reports`
- 在列表上方展示每个学校的状态卡

---

## 展示规则

### 状态值

- `success` -> `正常`
- `blocked` -> `被拦截`
- `error` -> `失败`

### 视觉

- `success` 使用绿色弱强调
- `blocked` 使用黄色弱强调
- `error` 使用红色弱强调

### 内容

每张状态卡显示：

- 学校名
- 当前状态
- 本轮条目数
- 错误信息（如果有）

---

## 风险与边界

1. 本次只接 `university_site`，不扩展到其他来源的 sync 状态
2. `Oxford` 当前会稳定显示为 `blocked`，直到后端补 browser fallback
3. 本次不改 Dashboard 的排序、筛选、已读、告警逻辑

---

## 验收标准

1. 总览页 `来源状态` 不再写死 `4/4`
2. 大学官网来源页能看到 Cambridge / Imperial / UCL / Oxford 四张状态卡
3. `Oxford` 明确显示 `被拦截`
4. 状态接口失败时，总览页仍可正常打开
