# 运营情报聚合面板设计规格文档

**日期：** 2026-05-19  
**范围：** 在现有 `xhs-sentiment-monitor` 基础上，把前端 Dashboard 从“小红书舆情面板”扩展为“运营工作台”，接入并组织 `小红书`、`UCAS`、`海外大学官网`、`媒体公众号` 四类来源的新增情报。  
**阶段：** 信息架构与交互结构设计，不进入实现细节。

---

## 背景与目标

当前项目的前端结构仍以“小红书舆情监控”视角为主，核心模块是情感分布、趋势、竞品对比和笔记列表。这套结构适合看舆情，但不适合作为国际教育内容运营的日常入口。

这次扩展的目标不是继续堆图表，而是把面板重新定义成 `运营工作台`：

1. 用户打开首页后的第一动作是 `看今天新增情报`
2. 信息按 `来源` 组织，而不是按情感图表或算法排序组织
3. 面板先帮助用户 `快速理解新增内容`，再引导用户 `去原文`
4. 首版要克制，优先把信息架构做清楚，而不是一开始就把选题池、待办、复杂联动全部塞进首页

---

## 核心设计决策

| 维度 | 决策 |
|------|------|
| 产品定位 | 运营工作台，而非纯舆情大屏 |
| 首页第一动作 | 看今天新增情报 |
| 首页主组织方式 | 按来源 |
| 默认阅读路径 | 看摘要 -> 判断价值 -> 去原文 |
| 首页骨架 | 左侧来源导航 + 中间卡片流 + 右侧轻辅助栏 |
| 总览页密度 | 轻量入口，不做深度阅读 |
| 摘要策略 | 总览页每条 1 句；来源详情页每条 2-3 句 |
| 大学官网组织方式 | 混合模式：总分区 + 重点学校直出 |
| 公众号范围 | 偏媒体与垂类号，而非自家号/竞品号优先 |

---

## 设计范围

本设计覆盖：

- 新首页信息架构
- 左侧来源导航层级
- 总览页结构
- 各来源详情页结构
- 情报卡片的字段和阅读节奏
- 统一来源模型需要支持的基础信息
- 空状态、同步状态、异常状态的产品行为

本设计不覆盖：

- 具体抓取实现
- 摘要生成 prompt 或模型选择
- 选题池、待办系统、发布系统的完整交互
- 跨来源主题聚类页
- 站内详情页替代原文
- 首版复杂图表和多维分析台

---

## 整体信息架构

### 全局骨架

首页和各来源页统一采用三栏骨架：

- **左侧导航**：来源入口，承担一级信息架构
- **中间主区**：当前来源的情报卡片流，是绝对主阅读区
- **右侧辅助栏**：轻量提示区，只做当天重点提示，不承载复杂交互

这个骨架的原则是：`来源导航稳定，中间阅读优先，右侧尽量克制`。

右侧辅助栏只保留：

- 今日重点条数
- 影响对象分布（如本科 / 硕士 / 申请季）
- 轻量同步提示

右侧辅助栏明确不放：

- 选题池
- 待办队列
- 复杂筛选
- 深度统计图表

这样可以避免首页从“情报入口”滑向“后台控制台”。

### 左侧来源导航

一级导航固定为：

- `总览`
- `小红书`
- `UCAS`
- `海外大学官网`
- `媒体公众号`

只有 `海外大学官网` 允许展开二级层，因为这是唯一既是一个来源、又天然内含多个重点对象的分区。

`海外大学官网` 的二级层为：

- `重点学校`
- `全部学校`
- `官方新闻 / 招生动态 / 奖学金`

其他来源不做二级导航：

- `UCAS` 是单一强来源，保持扁平
- `媒体公众号` 在内容区里轻筛账号/主题即可，不放到左侧做复杂层级
- `小红书` 更适合在内容区看讨论和放量话题，而不是在导航中继续细分

## 页面结构

### 1. 总览页

总览页不是“第二个首页大杂烩”，而是四类来源的轻量入口。

### 顶部轻指标

总览页顶部只放 3 个轻指标：

- `今日新增`
- `已读`
- `来源状态`

这里的 `来源状态` 用来表达各来源是否成功同步，属于运营值班所需的最小健康信息，不做技术诊断面板。

### 来源分区

总览页中间按来源依次展示四个分区：

- 小红书
- UCAS
- 海外大学官网
- 媒体公众号

每个来源分区只做以下事情：

- 显示该来源今天新增多少条
- 给出 2-3 条预览情报
- 每条预览情报露出 `1 句短摘要`
- 提供“进入该来源页”的动作

总览页不做的事情：

- 不堆满长列表
- 不展开完整摘要
- 不承载复杂筛选
- 不放运营待办和选题池

总览页的职责只有一个：`先扫全盘，再决定进入哪个来源深读`。

### 2. 小红书来源页

小红书来源页的角色是 `社区温度计`。

默认模块顺序：

1. 今日新增讨论卡片流
2. 上升关键词
3. 放量话题

这里的重点不是政策权威性，而是：

- 用户在讨论什么
- 哪些话题突然放量
- 社区情绪或关注点是否值得内容侧跟进

### 3. UCAS 来源页

UCAS 来源页的角色是 `高确定性官方信号源`。

默认模块顺序：

1. 政策更新
2. 时间节点变化
3. 服务 / 流程 / 申请系统变更

摘要语气要更接近“要点提炼”，而不是媒体式转述。这个来源页应该帮助运营快速回答：`今天有没有会影响申请节奏或内容判断的官方变化`。

### 4. 海外大学官网来源页

海外大学官网来源页采用混合模式，是整个面板里结构最复杂的一页。

默认阅读顺序：

1. 重点学校更新
2. 全部学校更新
3. 分类视角（官方新闻 / 招生动态 / 奖学金）

重点学校模块应该优先露出你长期关注的 3-5 所学校的新增内容，让首页逻辑和长期跟踪逻辑自然衔接。

这个来源页的卡片必须明确露出：

- 学校名
- 影响对象

否则运营无法快速判断内容优先级。

### 5. 媒体公众号来源页

媒体公众号来源页的角色是 `外部解读层`。

默认模块顺序：

1. 新文章卡片流
2. 轻量账号筛选
3. 轻量主题筛选

这里不建议一开始就做复杂导航层级，因为公众号阅读行为更像“先看内容，再决定是否关注账号”，不是先切管理树。

## 情报卡片设计

情报卡片是整个面板的核心单元。

### 总览页卡片

总览页的卡片字段为：

- `标题`
- `1 句短摘要`
- `来源/学校名`
- `影响对象`

这里的目的是让用户在 1-2 秒内判断“值不值得点进去”，所以总览页卡片必须短。

### 来源详情页卡片

来源详情页的卡片字段为：

- `标题`
- `2-3 句完整摘要`
- `来源/学校名`
- `影响对象`
- `去原文`

这张卡片必须直接回答三个问题：

1. 发生了什么
2. 为什么值得运营看
3. 它影响谁

### 时间信息

时间戳保留，但应弱化显示。它的作用是辅助判断“是不是今天的新内容”，而不是抢占主阅读节奏。

### 首版不放进卡片的字段

为了保持清晰，首版卡片先不放：

- 全文摘录
- 作者信息
- 复杂标签云
- 情感图表
- 多个运营动作按钮

默认保留一个强动作：`去原文`。

---

## 统一来源模型

虽然这次是结构设计，不进入实现，但首版必须围绕统一情报模型设计，否则四类来源很快会在前端分裂成四套 UI。

每条情报至少应被归一化为以下字段：

- `id`
- `source_type`
  - `xiaohongshu`
  - `ucas`
  - `university_site`
  - `wechat_media`
- `source_name`
  - 如 `UCAS`、`UCL`、`留学咖啡馆`
- `title`
- `summary_short`
- `summary_long`
- `impact_targets`
  - 如 `本科`、`硕士`、`申请季`
- `published_at`
- `collected_at`
- `original_url`
- `priority_hint?`
  - 可为空，用于右侧轻辅助栏或未来排序视图

这个统一模型的意义不是把所有来源“做得一样”，而是保证：

- 总览页可以复用同一套预览卡片
- 来源页可以共享统一卡片骨架
- 后续若要加“高优先级视图”或“选题池联动”，不需要重做前端数据契约

## 数据流与页面行为

### 数据流

首版推荐的数据流是：

1. 各抓取器分别采集不同来源内容
2. 后端将原始内容归一化为统一情报项
3. 摘要层为每条情报生成 `summary_short` 和 `summary_long`
4. 前端按 `source_type` 进行分组展示
5. 总览页只拉每个来源的少量预览
6. 来源页拉该来源的完整分页结果

这里故意不把“复杂聚类”和“主题聚合”放进第一版数据流，因为那会显著拉高实现复杂度，也偏离当前已经确认的来源导向结构。

### 页面行为

默认行为链路为：

1. 打开总览页
2. 快速看四类来源今天各新增了什么
3. 进入某个来源页
4. 阅读 2-3 句摘要
5. 点击 `去原文`

这条链路必须始终顺滑，任何附加模块都不应该打断它。

## 空状态、异常状态与同步状态

### 空状态

如果某个来源当天没有新增内容：

- 该来源仍然保留在导航中
- 总览页显示“今日暂无新增”
- 来源页给出空状态说明，而不是整块消失

原因是对运营来说，“今天没有更新”本身也是有效信息。

### 异常状态

如果某个来源同步失败：

- 总览页顶部 `来源状态` 应能提示有来源异常
- 对应来源页展示“最近一次同步失败”及最近成功时间
- 不在主阅读区塞技术错误堆栈

这既能保留运营值班感，也避免把技术诊断信息污染主界面。

### 摘要缺失

如果某条内容尚未成功生成摘要：

- 总览页应优先回退为标题预览，不阻塞整个分区
- 来源页可显示“摘要生成中”或简短占位文案

原则是：摘要是核心体验，但单条摘要失败不应该拖垮整页可用性。

---

## 首版不做的能力

为了保证 MVP 干净，以下能力明确放到后续阶段：

- 选题池联动
- 内容待办管理
- 跨来源主题聚合
- 算法优先级首页
- 情报详情页替代外链原文
- 重分析图表页
- 高密度后台表格

这一步的成功标准不是“全能”，而是：`用户每天愿意打开它看新增情报`。

---

## 验证方法

设计验收应围绕以下问题：

1. 用户是否能在 10 秒内理解左侧导航的来源结构
2. 用户是否能在总览页快速扫到四类来源今天的新增情况
3. 用户是否能在来源页仅凭摘要就判断是否需要看原文
4. 右侧辅助栏是否足够轻，不会抢走中间主区的注意力
5. 总览页和来源页之间是否形成明确分工：前者轻扫，后者深读

如果以上五点成立，这套结构就已经具备进入实现计划的条件。
